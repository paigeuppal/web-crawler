## Architecture

The system is composed of four independent modules that form a pipeline:

```
 quotes.toscrape.com
        │
        ▼
 ┌─────────────┐     HTTP GET (6s delay)     ┌─────────────┐
 │  crawler.py │ ──────────────────────────► │  Web Pages  │
 │  BFS queue  │ ◄── HTML response ───────── │  (HTML)     │
 └──────┬──────┘                             └─────────────┘
        │ {url: plain_text}
        ▼
 ┌─────────────┐
 │ indexer.py  │  tokenise → filter stopwords → record positions
 │             │  build _meta (doc_count, doc_lengths)
 └──────┬──────┘
        │ index.json
        ▼
 ┌─────────────┐
 │  search.py  │  find_pages (intersection) → rank_pages (TF-IDF)
 │             │  suggest (prefix / edit distance)
 └──────┬──────┘
        │
        ▼
 ┌─────────────┐
 │   main.py   │  build | load | print | find | stats | quit
 │  CLI shell  │
 └─────────────┘
```

**Data flow:**
1. `crawler.py` performs a breadth-first crawl, issuing one HTTP request every 6 seconds. It returns a `dict[url → plain_text]`.
2. `indexer.py` tokenises each page, filters stopwords, and builds the inverted index. Document lengths are stored in `_meta` for TF-IDF computation.
3. The index is serialised to `data/index.json` and can be reloaded without re-crawling.
4. `search.py` queries the index: `find_pages` intersects posting lists, then `rank_pages` scores each result with TF-IDF.

## Data Structures & Design Rationale

### Inverted Index - nested dict

```python
{
    "_meta": {
        "doc_count": 287,
        "doc_lengths": {"https://quotes.toscrape.com/": 412, ...}
    },
    "life": {
        "https://quotes.toscrape.com/": {"frequency": 3, "positions": [42, 198, 310]},
        "https://quotes.toscrape.com/page/2/": {"frequency": 1, "positions": [88]}
    }
}
```

A nested `dict` was chosen over a flat list or database for three reasons:
- **O(1) word lookup** - Python dicts are hash maps; looking up a word costs constant time regardless of index size.
- **O(1) per-page lookup** - the inner dict maps URL → stats, so checking whether a page contains a word is also O(1).
- **Direct JSON serialisation** - the structure maps directly to JSON without any transformation, making `build` and `load` trivial.

An alternative would be a sorted list of `(word, url, freq)` tuples, but this gives O(log n) lookup and makes intersection more complex.

### Stopwords - `frozenset`

```python
STOPWORDS: frozenset[str] = frozenset({"the", "a", "is", ...})
```

`frozenset` was chosen over `list` or `set` because:
- **O(1) membership test** - `word in STOPWORDS` is a hash lookup, not a linear scan.
- **Immutable** - signals clearly that the stopword list is a constant, not something to be modified at runtime.
- A `list` would give O(n) lookup, adding ~70 comparisons per token during indexing.

### BFS Queue - `list` used as a deque

The crawler uses a `list` as a FIFO queue (`pop(0)`). For very large crawls this is O(n) per dequeue. A production system would use `collections.deque` which gives O(1) at both ends. For the scale of this project (~300 pages) the difference is negligible, but this is a known trade-off.

## Algorithms & Complexity Analysis

### Crawling - O(P)

Where P = number of pages on the domain. Each page is visited exactly once (enforced by the `visited` set). URL deduplication via the set costs O(1) per lookup. Total time is dominated by network I/O and the politeness delay: `P × 6 seconds`.

### Index Building - O(T)

Where T = total number of tokens across all pages. Each token is processed once: lowercased, checked against `STOPWORDS` (O(1)), and inserted into the index (two O(1) dict writes). Total: O(T).

### Search (find_pages) - O(W × D)

Where W = number of query words, D = average number of pages per word's posting list. The algorithm intersects posting lists using Python set intersection (`&=`), which runs in O(min(|A|, |B|)) per pair. For W query words this is O(W × D).

### TF-IDF Ranking - O(C × W)

Where C = number of candidate URLs after intersection, W = number of query words. Each candidate is scored by iterating over query words and performing O(1) dict lookups. Sorting C results is O(C log C), but C is always small in practice.

### TF-IDF Formula

```
TF(term, doc)  = frequency(term, doc) / doc_length(doc)
IDF(term)      = log(1 + total_docs / docs_containing_term)
score(doc)     = Σ TF(term, doc) × IDF(term)  for each query term
```

IDF is smoothed with `log(1 + ...)` to prevent division-by-zero when a term appears in every document and to dampen the weight of very common terms. This is equivalent to the smoothed IDF variant used in scikit-learn's `TfidfTransformer`.

## Usage

Start the shell from the repository root:

```bash
python src/main.py
```

### Commands

| Command | Description |
|---|---|
| `build` | Crawl the website, build the index, save to `data/index.json` |
| `load` | Load a previously saved index from disk |
| `print <word>` | Print the full index entry (frequency + positions) for a word |
| `find <word> [word ...]` | List pages containing all words, ranked by TF-IDF score |
| `stats` | Show index statistics (page count, word count, top words) |
| `help` | Show available commands |
| `quit` | Exit the shell |

### Example session

```
> build
Starting crawl of https://quotes.toscrape.com/ …
  Crawled (1, queue: 12): https://quotes.toscrape.com/
  Crawled (2, queue: 23): https://quotes.toscrape.com/page/2/
  ...
Index saved to data/index.json  (4167 unique words).

> load
Index loaded from data/index.json  (4167 unique words).

> print life
  'life' appears in 14 page(s):

  URL       : https://quotes.toscrape.com/
  Frequency : 3
  Positions : [42, 198, 310]
  ...

> find good friends
  Found 3 page(s) (ranked by relevance):
    [0.0521]  https://quotes.toscrape.com/page/2/
    [0.0341]  https://quotes.toscrape.com/tag/friendship/
    [0.0187]  https://quotes.toscrape.com/page/7/

> find the
  'the' is a stopword and was excluded from the index.
  No pages found.

> find xyznotaword
  'xyznotaword' not found. Did you mean: ...?
  No pages found.

> stats
  Index statistics
  ────────────────────────────────
  Pages indexed   : 287
  Unique words    : 4167
  Total tokens    : 89,421
  Avg words/page  : 311

> quit
```

### Edge cases handled

| Input | Behaviour |
|---|---|
| `find` (no args) | Prints usage hint |
| `find xyznotaword` | Reports not found, offers suggestions |
| `find the` | Explains stopword exclusion |
| `find good friends` | Returns only pages containing **all** words |
| `print` (no args) | Prints usage hint |
| `print xyznotaword` | Reports word not in index |
| Network error during crawl | Logs error, continues to next URL |

## Testing

Run the full test suite with coverage:

```bash
pytest tests/ -v --cov=src --cov-report=term-missing
```

Run a specific test file:

```bash
pytest tests/test_crawler.py -v
```

| `indexer.py` | 20 | 100% |
| `main.py` | 69 | 100% |

| **Total** | **174** | **100%** |

### Testing strategy

- **Unit tests** - each function tested in isolation with mocked dependencies (network calls, file I/O)
- **Integration tests** - `cmd_build` and `cmd_load` tested end-to-end through the pipeline using `tmp_path`
- **Edge case tests** - empty queries, whitespace-only input, missing index file, network errors, duplicate URLs
- **Performance tests** - `build_index` benchmarked against large synthetic datasets to verify O(T) scaling
